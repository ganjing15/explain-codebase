NanoClaw Architecture

How a message travels from your chat app to an AI response — and back.

How It Works
Step 1 — Message arrives
🔔
New message
src/channels/lark.ts
A chat platform pings NanoClaw the instant someone sends a message — like a doorbell ringing the moment a visitor arrives. NanoClaw doesn't have to keep checking; it just waits and reacts.
webhook POST /webhook
🌐
Works with any platform
src/channels/registry.ts → registerChannel()
NanoClaw speaks every platform's language through adapters — like a universal translator. You can switch platforms without changing anything else about how NanoClaw works.
lark telegram
Step 2 — Who's allowed in?
🚦
Trigger check
src/index.ts (trigger check)
Only messages that start with the right password get processed — everything else is quietly ignored. Like a bouncer at the door who only lets in people who say the right phrase.
@Nano requiresTrigger
🪪
Sender check
src/sender-allowlist.ts → isSenderAllowed()
Checks whether the person who sent the message is on the approved list. Admins can restrict Nano to only respond to certain people, or block messages from unknown senders entirely.
allowlist drop mode
Step 3 — Line up & wait your turn
🗂️
Queue
src/group-queue.ts → enqueueMessageCheck()
Each group chat gets its own waiting line — messages are handled one at a time so conversations never get jumbled together. Two different groups can be handled at the same time, but within a group, it's always orderly.
GroupQueue per-group
💾
Saved for context
src/db.ts → storeMessage()
Every message is stored so Nano can remember the full conversation history. This is how it knows what was said earlier and can give answers that make sense in context.
SQLite messages table
Step 4 — AI thinks it over
🔒 Isolated Docker Container
📦
Fresh workspace
src/container-runner.ts → runContainerAgent()
For each reply, NanoClaw spins up a clean, throwaway environment — like giving the AI a fresh notebook every time, completely sealed off from everything else. When it's done, the notebook is shredded.
docker run --rm --network=host
🔑
Secret keeper
src/credential-proxy.ts
The AI only ever sees fake placeholder keys. A local middleman quietly swaps them for the real ones — so your actual secrets never leave your machine, even if the AI misbehaves.
host.docker.internal:3001 ANTHROPIC_BASE_URL
🧠
Claude reasons & acts
src/index.ts → runAgent()
Claude reads the full conversation, decides which tools to use — web search, running commands, reading files — and works through the problem step by step, like a smart assistant with a full toolbox.
claude (latest) tools
🔧
Skills loaded
container/skills/ + groups/{name}/CLAUDE.md
Before thinking, Nano loads its instructions and any installed skills — like reading a briefing packet before a meeting. Each group can have its own custom set of instructions and capabilities.
SKILL.md system prompt
Step 5 — Reply goes back
📤
Response sent
src/index.ts → channel.sendFiles()
The AI's reply — whether it's text, images, or files — is handed back to the same platform the message came from. You get the answer right where you asked.
.send_files.json sendMessage()
📎
Files & images
src/channels/lark.ts → sendFiles()
If the AI created an image or a file during its work, it uploads them to the platform and sends them as proper attachments — not just a wall of text with no download link.
image_key file upload
Step 6 — Scheduled tasks
⏰
Runs on a schedule
src/task-scheduler.ts → getDueTasks()
Nano can run tasks automatically at set times — daily reminders, weekly reports, or anything you've scheduled in advance. Like a programmable alarm clock that triggers a full AI response when it goes off.
cron scheduled_tasks
📬
Cross-group messages
src/ipc.ts (polls every 1s)
Groups and AI containers can send messages to each other by dropping small notes in a shared folder — like a mail slot between rooms. No direct connection needed; just write a note and walk away.
ipc/messages/ ipc/tasks/
Key Concepts
🔔
Concept
Webhook
▾
Like a doorbell — instead of NanoClaw constantly knocking on Lark's door asking "any new messages?", Lark rings NanoClaw the moment something arrives.

Webhooks are how the outside world notifies NanoClaw. Chat platforms send an HTTP POST request to NanoClaw's URL whenever someone sends a message. No polling, no delay — instant push delivery.

  • Instant delivery — NanoClaw is notified the moment a message is sent, not on a timer
  • Platform-agnostic — every platform (Lark, WhatsApp, Telegram) uses the same webhook concept
  • Verification — platforms include a secret token so NanoClaw can confirm the sender is legit
POST https://your-server.com/webhook/lark → lark.ts handleWebhook()
🐳
Concept
Docker (Isolated Containers)
▾
Like a disposable lunchbox — packed fresh for every meal, used once, then thrown away. Nothing from one meal contaminates the next.

Every time NanoClaw needs the AI to work on a reply, it creates a fresh isolated environment — a "container". When it's done, the container disappears. This means one conversation can never accidentally affect another, and nothing the AI does can permanently change the host machine.

  • Isolated — each conversation runs in its own bubble; no shared state
  • Throwaway — containers are deleted after use (--rm flag)
  • Safe — even if the AI runs bad code, it's contained
docker run --rm ... nanoclaw-agent:latest
🔑
Concept
Credential Proxy
▾
Like a hotel concierge — guests can request things that need the master key (unlocking a room, accessing storage), but they never touch the key themselves.

The AI container only has fake placeholder API keys. When it makes an API call, a small "proxy" running on the host intercepts it and swaps in the real keys before forwarding. This way, even if the AI were somehow compromised, your real credentials stay safe on the host — the container never sees them.

  • Zero exposure — containers hold fake keys like sk-placeholder
  • Transparent — the AI doesn't know the swap happens; it just works
  • Host-only — real keys only ever live on your machine
ANTHROPIC_BASE_URL=http://host.docker.internal:3001 # containers can't use "localhost" — host.docker.internal # is Docker's special hostname that points back to your Mac # credential-proxy.ts listens on port 3001 and swaps in real keys
🗂️
Concept
Per-Group Message Queue
▾
Like a single checkout lane per customer at a store — a second customer can be helped at the next lane, but you never have two clerks bumping into each other serving the same person.

Each group chat has its own queue. Messages wait in line and are processed one at a time per group. This prevents chaos — if two people message at the same moment, their replies won't get mixed up. But two different groups are handled in parallel with no waiting.

  • Ordered — messages in a group are always handled in order
  • Per-group — groups don't block each other
  • Backpressure — if a response takes a while, new messages for that group wait nicely
GroupQueue.enqueue(groupId, async () => runAgent(...))
📬
Concept
File-Based IPC (Inter-Process Communication)
▾
Like a mail slot in a door — one side drops a note through, the other side picks it up and acts on it. No direct connection needed.

The AI container and the host can't talk directly (they're isolated). Instead, they communicate by writing small JSON files to a shared folder. The host watches the folder and picks up any new files. This is how the AI can schedule tasks, send messages to other groups, or trigger actions on the host.

  • Simple & reliable — just files; no sockets, no ports to manage
  • Decoupled — sender and receiver don't need to be running at the same time
  • Auditable — you can inspect the files to see what was requested
// AI writes: {"type": "message", "chatJid": "...", "text": "Hello"} → /workspace/ipc/messages/msg_123.json // Host picks it up via fs.watch()
🌐
Concept
Channel Abstraction
▾
Like a universal power adapter — your laptop works in any country because the adapter handles the different plug shapes. NanoClaw works on any chat platform for the same reason.

Every chat platform works differently — Lark has its own API, Telegram has another, WhatsApp yet another. NanoClaw defines a single common interface ("channel") and then writes a thin adapter for each platform. The core system only talks to the common interface and never cares which platform it's actually on.

  • Pluggable — add a new platform by writing one adapter file
  • Consistent — the core always calls the same send(), sendFiles() methods
  • Independent — Lark bugs don't affect Telegram; each adapter is self-contained
// src/types.ts interface Channel { sendMessage(jid: string, text: string): Promise<void>; sendFiles?(jid, attachments, groupFolder): Promise<void>; // optional connect(): Promise<void>; isConnected(): boolean; }
🧩
Concept
Skills (Plugin System)
▾
Like an app store for your AI — install a skill and new superpowers appear. Uninstall it and they vanish, without touching anything else.

Skills are Markdown files that give Nano new capabilities. When a skill is installed, its instructions are injected into the AI's briefing before it starts thinking. The AI reads the skill and knows how to do new things — browse the web in a special way, generate documents, manage a specific workflow — without any code changes.

  • Zero-code — skills are plain Markdown files, not code
  • Per-group — different groups can have different skills installed
  • Composable — install multiple skills; they stack
container/skills/explain-codebase/SKILL.md → injected into system prompt groups/main/CLAUDE.md → group-specific instructions
Key Files & Directories
🚀 Entry Point
src/index.ts
Starts everything: registers channel adapters, wires up the scheduler, and listens for incoming webhooks.
🔔 Channel Adapters
src/channels/lark.ts, telegram.ts
One file per platform — each knows how to receive messages and send replies for its platform.
🗂️ Message Queue
src/group-queue.ts
Keeps per-group queues so messages are handled one at a time per conversation.
🤖 Container Runner
src/container-runner.ts
Spawns Docker containers, injects credentials, runs Claude, collects replies via runContainerAgent().
🔑 Credential Proxy
src/credential-proxy.ts
HTTP proxy on port 3001 — containers reach it via host.docker.internal:3001 and it swaps fake keys for real ones transparently.
⏰ Scheduler
src/task-scheduler.ts
Checks the database every 60s for due tasks and fires them as full agent runs.
🗄️ Database
src/store/ (SQLite)
Stores messages, registered groups, scheduled tasks, and conversation history.
📬 IPC Directory
/workspace/ipc/
Shared folder between host and containers. messages/ for outbound, tasks/ for scheduling.
🧠 Group Memory
groups/{name}/CLAUDE.md
Per-group instructions and memory. Each group gets a private persistent context.
🧩 Skills
container/skills/
Installed skills. Each is a Markdown file synced into the agent container and injected into the system prompt.
