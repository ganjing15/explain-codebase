NanoClaw Architecture

A personal Claude assistant — messages arrive from any chat platform, get routed to an isolated AI container, and replies flow back. Single Node.js process, multiple messaging platforms, zero secrets in containers.

Overview
How It Works
Step 1 — Message arrives
📨
Lark
src/channels/lark.ts
A message comes in from a Lark group chat. The channel adapter listens for new events and hands them to the core system.
✈️
Telegram
src/channels/telegram.ts
A message comes in from a Telegram bot. Same callback interface — the core never cares which platform it came from.
💬
Other channels
src/channels/registry.ts → registerChannel()
WhatsApp, Slack, Discord, and more can plug in. Any channel that registers itself at startup gets treated identically.
Step 2 — Stored & checked
🗄️
Saved to database
src/db.ts → storeMessage()
Every message is written to a SQLite database immediately, even if the AI won't respond to it. This builds up conversation history that gives the AI context.
🚪
Sender & trigger check
src/sender-allowlist.ts → isTriggerAllowed()
The system checks two things: is this sender allowed to talk to the AI, and does the message contain the trigger phrase (like "@Nano")? Non-trigger messages accumulate as context but don't wake the AI.
Step 3 — Queue & wait
🚦
One lane per group
src/group-queue.ts → GroupQueue.enqueueMessageCheck()
Each chat group has its own queue. If the AI is already busy replying to that group, the new message waits — they won't get served by two AI instances at once.
📡
Pipe to active container
src/group-queue.ts → GroupQueue.sendMessage()
If the AI container is already running for this group, the new message is piped in through stdin — no need to start a new container. Saves time and cost.
Step 4 — Agent container starts
🎒
Packed & launched
src/container-runner.ts → runContainerAgent()
A fresh throwaway container is packed with the group's personal folder and a read-only view of the project. It's launched, used, and discarded — nothing leaks between runs.
🔑
Credentials injected
src/credential-proxy.ts → startCredentialProxy()
The container gets a fake API key. When it tries to call Claude, the traffic is intercepted by a local proxy that swaps in the real secret — the container itself never sees it.
🔒 Inside the agent container
🤖
AI reads the conversation
container/agent-runner/src/index.ts
Claude reads all the pending messages, loads the group's memory file, and any relevant skills. It can browse the web, read files in its workspace, run bash commands.
📝
Response written to IPC
container/agent-runner/src/index.ts → OUTPUT_START_MARKER
When done, Claude writes its reply to stdout using sentinel markers. The host process reads the stream and extracts the response.
Step 5 — AI thinks & acts
💭
Output streamed back
src/index.ts → processGroupMessages()
As the AI produces results, they stream back in real time. The typing indicator stays on while the AI works, so the chat feels responsive.
📎
Files sent if needed
src/index.ts → channel.sendFiles()
If the AI produced a file (a chart, a document, an image), it gets sent as an attachment via the same channel that received the original message.
Step 6 — Reply sent back
✅
Reply delivered
src/router.ts → findChannel() → channel.sendMessage()
The reply goes back through the same channel it came in on. The router matches the group ID to the right channel — Lark replies go to Lark, Telegram to Telegram.
📌
Cursor advanced
src/index.ts → saveState()
After a successful reply, a bookmark is saved so the AI won't re-process the same messages on restart. If there was an error before a reply was sent, the bookmark stays put so the message gets retried.
Key Concepts
🔌
Channel abstraction
▾
Like a universal power adapter — same laptop, different plugs for every country. The core system only knows how to talk to "a channel", and each country (platform) provides its own plug.

NanoClaw supports multiple messaging platforms without its core logic knowing anything about any of them. Each channel self-registers at startup by calling registerChannel(), and the registry hands out factories when needed.

  • Self-registration — Channels register themselves at startup by importing and calling registerChannel(); the core never lists platforms explicitly
  • JID-based routing — Each chat has a unique JID (e.g. 120363@g.us for WhatsApp, lark:oc_... for Lark); ownsJid() is how the right channel is found for a reply
// src/channels/registry.ts registerChannel(name: string, factory: ChannelFactory): void getChannelFactory(name: string): ChannelFactory | undefined getRegisteredChannelNames(): string[]
🚦
Per-group queue
▾
Like a single checkout lane per customer — no two cashiers serve the same person at once. Every chat group gets its own lane, and messages wait in line within it.

Without a queue, two messages arriving quickly from the same group could spawn two containers simultaneously, leading to duplicate replies and race conditions. The GroupQueue ensures only one container runs per group at a time.

  • Stdin piping — If a container is already running when a new message arrives, the message is piped in via stdin rather than spawning a new container
  • Idle timeout — Containers stay alive for 30 minutes of inactivity, then stdin is closed — the next message will cold-start a fresh container
  • Global cap — MAX_CONCURRENT_CONTAINERS limits total parallel containers across all groups, not just per-group
// src/group-queue.ts GroupQueue.enqueueMessageCheck(groupJid: string): void GroupQueue.sendMessage(groupJid: string, text: string): boolean GroupQueue.notifyIdle(groupJid: string): void GroupQueue.closeStdin(groupJid: string): void
📦
Container isolation
▾
Like a disposable lunchbox — packed fresh for each meal, used once, then thrown away. What's in one box never touches another box, and nothing lingers.

Every agent run gets a throwaway Linux VM. When the agent finishes (or times out), the container is deleted. This means a misbehaving agent can't read other groups' files, and any mess it makes is cleaned up automatically.

  • Selective mounts — Group folder is read-write; project root is read-only; the agent can't write outside its group directory
  • Mount security — Custom mounts defined by a group are validated to prevent path traversal attacks
  • Runtime abstraction — Works with Docker or Apple Container; swap via CONTAINER_RUNTIME_BIN env var
// src/container-runner.ts runContainerAgent(group, input, onRegister, onOutput): Promise<ContainerOutput> // Mounts: // /workspace/group → groups/{folder}/ (read-write) // /workspace/project → project root (read-only) // /workspace/ipc → data/ipc/{folder} (read-write)
🔑
Credential proxy
▾
Like a hotel concierge with the master key — you make a request, and the concierge uses the master key on your behalf. You never touch the key yourself.

Containers receive a placeholder ANTHROPIC_API_KEY and their ANTHROPIC_BASE_URL points to localhost:3001 instead of Anthropic's servers. The proxy intercepts every request and injects real credentials before forwarding.

  • Two auth modes — API key mode (injects x-api-key header) or OAuth mode (intercepts the CLI's token exchange request)
  • Keychain integration — OAuth tokens are read from macOS Keychain via security find-generic-password
  • Token refresh — Refreshes OAuth tokens automatically when less than 1 hour remains
  • Bind address — On Linux, binds to the container's host gateway IP so containers can reach it; on macOS it binds to 0.0.0.0
// src/credential-proxy.ts startCredentialProxy(port: number, bindHost: string): Promise<Server> detectAuthMode(): AuthMode // 'api-key' | 'oauth' // Container env vars: // ANTHROPIC_BASE_URL=http://host-gateway:3001 // ANTHROPIC_API_KEY=placeholder
📬
File-based IPC
▾
Like a mail slot between two rooms — the agent drops a note through the slot, and the host picks it up and acts on it. No direct connection needed between them.

Containers can't call back to the host directly, but they can write JSON files to their IPC directory, which the host polls every second. This lets agents trigger actions like sending a message to a different group, registering a new group, or scheduling a task.

  • Outbound messages — Agents write to ipc/messages/ to send a message to any registered group, not just their own
  • Task scheduling — Agents write to ipc/tasks/ to schedule future tasks — this is how the AI can set its own reminders
  • Group registration — Agents can register entirely new groups by writing registration JSON files
// src/ipc.ts startIpcWatcher(deps: IpcDeps): void // Polls: data/ipc/{folder}/tasks/*.json // Message format: { type: 'message', chatJid: '...', text: '...' }
⏰
Task scheduler
▾
Like a programmable alarm clock that can do anything when it rings — not just beep, but run a full AI conversation on your behalf at 9am every weekday.

The scheduler runs a loop every 60 seconds, checking the database for tasks whose next-run time has passed. It fires them by spawning the same container pipeline as a normal message, so scheduled tasks have full AI capabilities.

  • Persisted in SQLite — Tasks survive process restarts; next_run is recomputed from the cron expression on each poll
  • 60-second resolution — Tasks fire within one minute of their scheduled time, not exactly on the second
// src/task-scheduler.ts startSchedulerLoop(deps: SchedulerDependencies): void computeNextRun(task: ScheduledTask): string | null // Task schema (SQLite): // id, group_folder, prompt, schedule_type, schedule_value, next_run, status
Key Files
🎛️ Orchestrator
src/index.ts
The main event loop. Polls for new messages, checks triggers, drives the queue, coordinates all subsystems on startup.
🔌 Channel registry
src/channels/registry.ts
Minimal plugin registry. Channels call registerChannel() at import time; the orchestrator iterates whatever registered itself.
📦 Container runner
src/container-runner.ts
Spawns agent containers with the right mounts, credentials, and IPC directories. Parses streamed output using sentinel markers.
🔑 Credential proxy
src/credential-proxy.ts
HTTP proxy on port 3001. Intercepts container API calls and injects real credentials. Handles both API key and OAuth auth modes.
🚦 Group queue
src/group-queue.ts
Per-JID serialization of agent runs. Prevents duplicate containers, pipes new messages to running containers, enforces global concurrency limit.
📬 IPC watcher
src/ipc.ts
Polls the IPC directories every second. Processes task files written by containers — sending messages, scheduling tasks, registering groups.
🗄️ Database
src/db.ts
SQLite via better-sqlite3. Stores messages, sessions, registered groups, router state, and scheduled tasks. All reads/writes are synchronous.
⏰ Scheduler
src/task-scheduler.ts
60-second loop that fires due tasks. Supports cron expressions (via cron-parser) and one-shot timestamps.
🤖 Agent runner
container/agent-runner/src/index.ts
Runs inside the container. Invokes the Claude CLI with the conversation prompt, streams results back with sentinel markers, sends files via .send_files.json.
📁 Group folders
groups/{name}/
Each registered group gets an isolated directory mounted into its container. Contains CLAUDE.md (memory), logs, and any files the agent creates.
